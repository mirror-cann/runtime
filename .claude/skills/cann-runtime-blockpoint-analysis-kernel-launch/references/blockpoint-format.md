# 断点记录格式模板

## 标准格式

每遇到一个资料缺失或不足的问题，按以下格式记录：

```
+---------------------------------------------------------------------+
| 断点 #N                                                              |
| 问题类型：Runtime 缺陷 / 外源知识缺失 / 体验问题                        |
| 发生步骤：Step X - 步骤名称                                          |
| 问题描述：我在尝试做什么，遇到了什么障碍                                |
| 查找过程：我在哪些文件中查找过（列出具体路径）                           |
| Runtime 文档中的相关内容：有/无（若有，覆盖了什么，遗漏了什么）          |
| 示例代码中的相关内容：有/无（若有，覆盖了什么，遗漏了什么）              |
| 缺失资料：具体缺少哪个文档 / 哪个 API 的说明                           |
| 影响程度：阻塞点（真正阻断开发，无法继续）/ 优化点（可解决但过程复杂）    |
| [推测]：根据类似框架经验的猜测（标注为推测，不计入产出）                 |
| 期望补充：理想情况下该有什么资料，建议由哪方补充                        |
+---------------------------------------------------------------------+
```

## 字段说明

### 问题类型

| 类型 | 含义 | 适用场景 |
|------|------|---------|
| **Runtime 缺陷** | 问题完全在 Runtime 仓库责任范围内 | API 文档缺失、示例不完整、错误码未说明等 |
| **外源知识缺失** | 非 Runtime 仓库职责范围的知识缺失 | AscendC 核函数编写、编译产物格式、tiling 参数结构等外部组件知识。**记录但不计入 Runtime 整改项** |
| **体验问题** | 不阻塞功能，但影响开发效率或学习体验 | 文档组织混乱、缺少导航、术语不一致等 |

### 影响程度

| 等级 | 含义 | 判定标准 |
|------|------|---------|
| **阻塞点** | 真正阻断开发，无法继续 | 关键 API 无任何文档、必要步骤完全不知如何操作，最终无法解决 |
| **优化点** | 可解决但过程复杂 | 有部分信息但不完整，需从示例代码反推或大量查找才能解决，最终能解决 |

### 查找过程

必须列出实际查找过的文件路径，例如：
- `docs/api_reference.md` — 未找到相关内容
- `docs/quick_start.md` — 第 3 节提到了但不完整
- `example/kernel/0_launch_kernel/main.cpp` — 有使用但无注释说明

### 期望补充

明确指出：
1. 理想情况下应该有什么资料
2. 建议放在哪个位置（如 `docs/api/memory.md`）
3. 是 Runtime 仓库应补充，还是属于外部组件的文档范畴

## 外源知识缺失的特别处理

对于 AscendC 与 Runtime 交界区域的问题，需要额外记录：

1. **Runtime 文档覆盖情况**：在 docs/ 和 example/ 中查找了哪些文件，找到了什么
2. **用户提供情况**：是否已向用户求助获取该知识，用户提供了什么
3. **缺失描述**：开发者在此处需要哪些外部知识才能继续
4. **处理方式**：向用户求助获取外源知识，不计入 Runtime 整改项

## 示例

```
+---------------------------------------------------------------------+
| 断点 #5                                                              |
| 问题类型：Runtime 缺陷                                                |
| 发生步骤：Step 5 - 编写 Runtime 调用框架代码                          |
| 问题描述：尝试使用 <<<>>> 内核调用符调用核函数，                        |
|          但文档中未完整说明三个参数（blockDim, l2ctrl, stream）的含义   |
| 查找过程：                                                            |
|   - docs/zh/02_dev_guide/Kernel加载与执行.md — 有简单示例但参数说明不完整  |
|   - docs/zh/01_quick_start/Runtime_programming_model.md — 有使用但无详细解释        |
|   - example/kernel/0_launch_kernel/main.cpp — 有使用但参数是硬编码的   |
| Runtime 文档中的相关内容：有语法示例，但参数含义不清晰                   |
| 示例代码中的相关内容：有使用，可反推参数顺序                            |
| 缺失资料：<<<>>> 内核调用符的完整参数说明                              |
| 影响程度：优化点（可从示例代码反推，但不确定是否正确）                   |
| [推测]：类似 CUDA 的 <<<gridDim, blockDim, sharedMem, stream>>>       |
| 期望补充：                                                            |
|   1. 在 docs/zh/02_dev_guide/ 中补充 <<<>>> 语法的完整说明               |
|   2. 包含每个参数的含义、取值范围、示例                                 |
+---------------------------------------------------------------------+
```
