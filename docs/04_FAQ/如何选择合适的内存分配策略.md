# 如何选择合适的内存分配策略

## 问题现象描述

**现象1：内存分配性能不佳**

内存分配速度慢、访问性能低，选择的页类型不适合业务场景。

典型场景示例：
```c
// 小块内存却配置了HUGE_ONLY策略，实际占用远超需求
aclrtMalloc(&devPtr, 1024, ACL_MEM_MALLOC_HUGE_ONLY);  // 实际占用 2MB（大页对齐）
```

**现象2：内存申请失败**

配置特定策略后内存申请失败，如强制使用大页但大页内存不足。

**现象3：P2P场景性能不佳**

跨Device复制场景使用普通策略而非P2P策略，性能不佳。

## 可能原因

1. **普通页 vs 大页选择不当**：不理解普通页和大页的性能特性、容量占用差异。
2. **P2P 场景未使用对应策略**：跨 Device 复制场景使用普通策略，性能不佳。
3. **频繁申请释放内存**：未预分配复用，导致性能损耗。

## 处理步骤

### 原因1：普通页 vs 大页选择不当

**解决方法**：
- 理解页类型特性：
  - **普通页（4K）**：适合小块、短生命周期、数量多的内存申请，减少对齐浪费
  - **2M大页**：适合大块、长期驻留、频繁访问的内存，减少页表项、扩大TLB覆盖
  - **1G大页**：适合超大块、长期驻留、性能敏感的内存（部分产品支持）
- 按数据大小选择策略：
  - ≤1M：使用 ACL_MEM_MALLOC_NORMAL_ONLY 或 HUGE_FIRST（自动降级为普通页）
  - >1M：使用 ACL_MEM_MALLOC_HUGE_FIRST（优先大页）
- 了解 HUGE_FIRST 机制：申请大小 ≤1M 时，即使配置 HUGE_FIRST 也使用普通页

策略选择示例：
```c
// 小块内存（≤1M）：普通页或 HUGE_FIRST（自动降级）
size_t smallSize = 1024 * 100;  // 100KB
aclrtMalloc(&ptr1, smallSize, ACL_MEM_MALLOC_NORMAL_ONLY);      // 普通页，占用约100KB
aclrtMalloc(&ptr2, smallSize, ACL_MEM_MALLOC_HUGE_FIRST);       // 自动使用普通页

// 大块内存（>1M）：大页优先
size_t largeSize = 1024 * 1024 * 10;  // 10MB
aclrtMalloc(&ptr3, largeSize, ACL_MEM_MALLOC_HUGE_FIRST);       // 大页，占用约10MB（对齐到2M）
aclrtMalloc(&ptr4, largeSize, ACL_MEM_MALLOC_HUGE_ONLY);        // 仅大页，不足时报错

// 强制大页场景：对大页性能有强依赖
aclrtMalloc(&ptr5, largeSize, ACL_MEM_MALLOC_HUGE_ONLY);        // 大页不足时报错，尽早暴露问题
```

### 原因2：P2P 场景未使用对应策略

**解决方法**：
- 理解 P2P 场景：跨 Device 数据复制，内存需支持跨 Device 访问
- 使用 P2P 策略：选择带 P2P 后缀的策略（HUGE_FIRST_P2P、HUGE_ONLY_P2P、NORMAL_ONLY_P2P）
- 使内存属性匹配访问路径：P2P 策略确保内存属性与跨 Device 访问匹配

P2P 策略示例：
```c
// 跨 Device 复制场景
aclrtSetDevice(0);
aclrtDeviceEnablePeerAccess(1, 0);  // 开启 Device 0→1 的 P2P

aclrtSetDevice(1);
aclrtDeviceEnablePeerAccess(0, 0);  // 开启 Device 1→0 的 P2P

// Device 0 上申请用于 P2P 的内存
aclrtSetDevice(0);
void* dev0Mem = nullptr;
aclrtMalloc(&dev0Mem, size, ACL_MEM_MALLOC_HUGE_FIRST_P2P);  // P2P 策略

// Device 1 上申请用于 P2P 的内存
aclrtSetDevice(1);
void* dev1Mem = nullptr;
aclrtMalloc(&dev1Mem, size, ACL_MEM_MALLOC_HUGE_FIRST_P2P);  // P2P 策略

// 跨 Device 复制
aclrtMemcpy(dev1Mem, size, dev0Mem, size, ACL_MEMCPY_DEVICE_TO_DEVICE);
```

### 原因3：频繁申请释放内存

**解决方法**：
- 预分配内存池：初始化阶段预分配固定大小的内存
- 业务侧复用：避免频繁调用 aclrtMalloc/aclrtFree
- 减少性能损耗：预分配复用可显著提升性能

内存池示例：
```c
// 初始化阶段预分配内存池
#define POOL_SIZE 10
#define MEMORY_BLOCK_SIZE (1024 * 1024)  // 1MB

void* memoryPool[POOL_SIZE];
bool memoryPoolUsed[POOL_SIZE] = {false};

void InitMemoryPool() {
    for (int i = 0; i < POOL_SIZE; i++) {
        aclrtMalloc(&memoryPool[i], MEMORY_BLOCK_SIZE, ACL_MEM_MALLOC_HUGE_FIRST);
        memoryPoolUsed[i] = false;
    }
}

// 业务中申请内存（从池中获取）
void* AllocateFromPool() {
    for (int i = 0; i < POOL_SIZE; i++) {
        if (!memoryPoolUsed[i]) {
            memoryPoolUsed[i] = true;
            return memoryPool[i];
        }
    }
    return nullptr;  // 池已满
}

// 业务中释放内存（归还到池）
void ReleaseToPool(void* ptr) {
    for (int i = 0; i < POOL_SIZE; i++) {
        if (memoryPool[i] == ptr) {
            memoryPoolUsed[i] = false;
            return;
        }
    }
}

// 清理阶段释放池
void CleanupMemoryPool() {
    for (int i = 0; i < POOL_SIZE; i++) {
        if (memoryPool[i] != nullptr) {
            aclrtFree(memoryPool[i]);
        }
    }
}
```
